# 260519学习笔记

## AI基础

### Frameworks框架

> 框架是系统边界的表达，不是智能本身。先理解工作流，再决定用不用框架。

Frameworks 用来解决的是工程组织问题。先画清楚输入、状态、工具、输出、评估和失败路径，再看哪些部分值得交给框架。

- 简单链路先保持简单：一次能完成的任务，不一定需要复杂 Agent 框架。
- 长流程需要显式状态：多步任务、工具调用、人工确认、失败恢复，应该有可查询的状态，而不是只靠聊天历史。
- 框架要能退出：如果某个框架很难换模型、换向量库、换部署方式，长期成本会很高。

**知识点**：

- LangChain：最常见的 LLM 应用开发框架之一，覆盖模型接入、prompt、工具调用、retriever、agent、output parser 等模块。
- LangGraph：更偏向工作流和状态机。它把 Agent 或多步骤任务表示成 graph：节点负责执行动作，边负责控制流，state 负责记录过程。
- OpenAI Agents SDK：用来构建带工具、handoff交接、guardrails限制条件、tracing追踪的 Agent 应用。可以用它组织：
  - Agent 的指令和工具
  - 多 Agent 之间的 handoff
  - 工具调用和结构化输出
  - guardrails 和运行时追踪
- DSPy：把 prompt / LM pipeline 写成可优化的程序。先定义清楚，再用 optimizer 改进 prompt 或调用策略。
- Hermes：面向工具调用和结构化输出的模型 / agent 生态，能稳定地产生所需的结构化输出和工具调用格式。
- Learning Agent：指系统可以从反馈、日志、评估结果或用户修正中改进。但是需要先进评估闭环，再进入生产系统。

**Frameworks 与 AI x Web3**

Context 负责把模型能力接到产品流程里：读取上下文、调用工具、生成结构化动作、记录 trace、进入 eval。

比较稳的分工是：

- AI Framework 管理 prompt、tools、state、eval 和 trace。
- Web3 基础设施管理账户、签名、合约、交易和链上状态。
- 产品层定义用户目标、权限边界、确认流程和失败处理。
- 层级越清楚，后续越容易做权限控制、prompt injection 防护和审计。

### Vibe Coding氛围编程

> AI Coding 的核心不是自动生成代码，而是把工程反馈循环压短。

AI Coding Agent 已经能读代码、改文件、跑测试、解释错误、生成 PR，这会明显改变开发速度。但作为 builder，你需要更清楚地管理 repo、任务、上下文、测试和提交边界。如果反馈循环里没有测试、审查和版本控制，速度只会放大混乱。

- 任务要小：越具体、越有边界，Agent 越容易产出可审查结果。
- 上下文要准：让 Agent 看对文件、设计约束和错误输出，比写长篇需求更重要。
- 验证要硬：运行测试、类型检查、构建、截图或日志，比“看起来对”可靠。

**知识点**：

- Vibe Coding：一种快速和 AI 共同探索代码的方式。先描述目标，让 Agent 生成方案或 patch，再通过运行、检查、追问和迭代收敛。适合搭原型、修小 bug、补测试、写脚本、重构局部模块、解释陌生代码，但不适合无边界地“重写整个项目”
- Claude Code / Codex CLI：把 AI Agent 放进本地开发环境，让它可以读文件、编辑文件、运行命令、理解测试输出。
- Model / API Config：AI Coding 不是只看“哪个模型最强”。你还要管理模型、API key、上下文窗口、工具权限、代理设置、成本和日志。好的配置应该能被团队复用，而不是每个人在本机随手调。常见问题包括：
  - 本地和 CI 用的模型不一致
  - API key 泄漏到日志或提交里
  - 上下文太长导致成本失控
  - Agent 使用了不合适的工具权限
  - 输出风格和项目规范不一致
- OpenCLI：把网站、浏览器会话、Electron 应用和本地二进制工具统一成可被命令行调用的接口，让 AI Agent 可以通过更稳定的命令去发现、学习和执行工具。
- GitHub / gh：AI Coding 工作流里的协作边界。Agent 可以帮你看 issue、生成 branch、读 PR diff、写提交信息、整理 review，但版本控制仍然是人类审查的关键线。让 Agent 多做局部 patch，少做不可追踪的大改动。每次改动后至少看：git diff、修改文件列表、测试结果、是否包含不该提交的密钥、日志、构建产物。
- CLI / Script：把 Agent 输出变成真实命令和脚本。开发任务不需要完整应用界面时，先写 CLI 或脚本更快。
- Repo Workflow：把 AI Coding 放进正常工程流程：issue、branch、commit、test、review、merge、release。比如：从 issue 提炼任务边界、搜索相关文件、生成最小 patch、跑测试并解释失败、补充 changelog 或文档、写 PR summary 和验证说明。

**Vibe Coding 与 AI x Web3**

Vibe Coding 可以显著加快探索速度，尤其适合 hackathon 和早期原型，但高风险动作必须经过审查、模拟和多方确认。

### MCP模型上下文协议

> 模型通过明确协议访问被授权的上下文和工具。

MCP 把模型和外部工具、数据源、应用上下文之间的连接标准化。

- 工具要有架构：没有 schema，模型调用工具就会变成自然语言猜参数。
- 权限要在协议外也成立：协议能描述能力，但真正的授权、审计和隔离仍要由系统实现。
- 错误要可传递：工具失败、超时、权限不足，必须明确返回，而不是让模型猜。

**知识点**：

- MCP Server：提供能力的一侧。
- MCP Client：连接模型和 MCP server 的一侧，比如桌面应用、IDE、Agent runtime 或聊天客户端。应该让用户知道：当前连接了哪些 server，模型能调用哪些工具，调用前是否需要确认。
- Tool Schema：描述工具名字、用途、参数、返回值和约束。它是模型正确调用工具的关键。
-Permission：权限的许可，至少要区分：只读/写入、临时授权/长期授权、是否需要用户确认、是否能访问敏感信息、是否会产生外部副作用、是否可撤销和审计。

**MCP 与 AI x Web3**

MCP 作为 Agent 连接链上工具和开发工具的接口层，是标准化工具入口，例如读取区块浏览器、查询合约文档、调用 RPC、生成交易草稿、读取项目 issue。

## Web3基础

### Wallet钱包

> 钱包是用户意图进入链上执行之前的最后一道确认界面。

只把钱包当成 SDK 接入，会很容易忽视安全和体验问题。在产品里，钱包交互至少要分清三类动作：

- 连接钱包：应用读取地址和网络，不等于可以动用资产。
- 签名消息：用户证明自己控制某个地址，可能用于登录、授权或创建订单。
- 发送交易：用户请求改变链上状态，可能转账、授权 token、调用合约或支付 gas。

钱包体系里，控制权来自私钥或智能账户规则。应用可以请求连接和签名，但不应该假设自己拥有用户账户。

- 连接不是授权资产：读取地址和发起交易是两层完全不同的权限。
- 签名必须可解释：用户需要知道签名目的，而不是只看到十六进制数据。
- 网络是上下文：同一个地址在不同链上可能有不同资产和状态。

**知识点**：

- EOA（Externally Owned Account）：由私钥控制的账户。它可以签名消息、发起交易、支付 gas、调用合约。大多数浏览器钱包和移动钱包默认给用户创建的都是这类账户。
- Mnemonic：一组单词，用来恢复钱包里的私钥。它的存在是为了让用户更容易备份，但也让钓鱼攻击更容易伪装成“输入助记词恢复账户”。
- Transaction：转账，或调用合约方法。钱包会让用户确认交易内容、链、合约地址、gas 和可能的资产变化。应用侧要尽量把这些信息解释清楚。点击按钮只是发起请求；交易还要经过钱包确认、签名、广播、打包和执行。每一步都可能失败。
- Gas：链上执行成本，用来支付网络执行和存储资源，既影响用户付费，也影响交易是否能被及时处理。用户发交易时通常需要支付 gas fee；如果 gas 估算不准、网络拥堵或余额不足，交易可能失败或卡住。
- Explorer：区块浏览器，可以查看地址、交易、合约、事件、token 转移和执行状态。对用户来说，它是确认交易是否成功的工具；对开发者来说，它是排查失败、核对合约和追踪状态的入口。


**Wallet 与 AI x Web3**

AI Agent 想要进入链上执行，最终会碰到钱包边界。它可以帮助解释交易、准备参数、检查风险、生成操作计划，但签名和权限不能被随意交给模型。

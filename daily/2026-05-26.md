## Notes

- Smart Contract 是部署在链上的程序，合约状态由链维护，调用结果需要通过交易或读取调用来观察。
- 普通转账主要改变账户余额；合约调用可能执行函数、修改状态、触发 event、转移资产或 revert。
- Read-only call 通常不改变链上状态，不需要 gas 支付；state-changing transaction 需要签名、gas，并可能失败。
- Event 是合约执行时写入日志的结构化记录，常用于前端展示、索引和链上行为追踪。
- Revert 表示交易执行中断，状态回滚，但用户可能仍然支付已消耗的 gas。

## Smart Contract Test Checklist

For each contract function, check:

- Expected successful path.
- Permission or ownership requirements.
- Invalid input handling.
- Revert reason or failure condition.
- State changes after execution.
- Event emission.
- Token or ETH balance changes.
- Boundary values.
- Interaction with external contracts, if any.

For AI x Web3 products, additionally check:

- Whether the agent is only preparing transaction data or actually requesting a wallet action.
- Whether the user can inspect contract address, method name, parameters, network, gas estimate, and asset impact.
- Whether risky actions require explicit human confirmation.
- Whether transaction hash and result are recorded for audit.

我把普通转账和合约调用区分开了：普通转账主要改变余额，而合约调用可能执行函数、修改合约状态、触发事件、转移资产，也可能因为权限、参数或业务条件不满足而 revert。对开发者来说，只知道“交易成功或失败”不够，还需要通过 event、交易回执、explorer 和测试来确认具体行为。

结合 AI x Web3，Agent 可以帮助用户解释合约调用、准备参数、检查风险和记录交易，但不应该替用户自动确认签名或广播高风险交易。后续做 Hackathon 原型时，交易确认界面必须展示合约地址、方法名、参数、网络、gas 估算和可能的资产变化。
